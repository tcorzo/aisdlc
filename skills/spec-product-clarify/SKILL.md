# spec-product-clarify（R1：Brainstorming）

## 概览

目标：把模糊需求推进到**可评审、可追溯、可进入 PRD/原型阶段**的“方案决策文档”`requirements/solution.md`。核心手段是：**一次一问（优先选择题）→ 每轮回写 `raw.md` 澄清记录 → 2–3 个差异明显的备选方案 → 推荐决策 + 风险验证计划**。

**REQUIRED SUB-SKILL:** 必须先使用 `spec-context` 定位 `FEATURE_DIR`，否则禁止读写任何 `requirements/*.md`。

## 何时使用 / 不使用

- **使用时机**
  - 已经存在对应 spec 分支与 spec pack，需要从 `{FEATURE_DIR}/requirements/raw.md` 出发做 R1（澄清+方案），输出 `{FEATURE_DIR}/requirements/solution.md`。
  - 需求信息不足/口径不清/边界不明，需要通过对话澄清并形成决策证据链。
- **不要用在**
  - 还没有 spec pack（先用 `spec-init`）。
  - 只想“直接写 PRD/原型”但 `solution.md` 还不存在（先完成本技能再进入后续阶段）。

## 核心原则（硬要求）

- **一次一问**：每条消息只问 1 个问题；同一主题需要深入就拆成多轮。
- **优先选择题**：给出 A/B/C 选项，并**先给你的推荐选项**（1–2 句理由）。
- **坚决 YAGNI**：优先删减范围，把不确定性留到“待确认问题 + 验证计划”，而不是硬补功能。
- **探索备选方案**：必须提出 **2–3 个差异明显**的备选方案（机制不同，而不是换皮）。
- **渐进验证**：当你认为理解足够时，用 **200–300 字**一段段呈现方案/设计，每段后问“是否正确？”确认后再继续。
- **增量回写**：每轮问答后，都要把“问题 + 推荐选项 + 用户选择/回答 + 一句话结论 + 遗留歧义（如有）”追加到 `raw.md` 的 `## 澄清记录`。
- **停止机制**：满足 DoD / 用户要求停止 / 边际收益过低时立刻收束，把剩余未知转入“待确认问题 + 验证计划”。

## 快速参考（产物与路径）

- **输入（必须）**：`{FEATURE_DIR}/requirements/raw.md`
- **增量证据链（必须更新）**：`{FEATURE_DIR}/requirements/raw.md` 的 `## 澄清记录`
- **输出（必须）**：`{FEATURE_DIR}/requirements/solution.md`
- **正文结构**：对齐模板 `.aisdlc-cli/templates/spec/product-solution-template.md`

## 实施步骤（Agent 行为规范）

### 0) 上下文定位（必须先做，否则停止）

按 `spec-context` 的唯一做法运行 `Get-SpecContext`，拿到并回显 `FEATURE_DIR=...`。

- **失败就停止**：如果 `Get-SpecContext` 报错，不得继续提问/生成任何文档内容。

### 1) 读取最小必要上下文（渐进式披露）

- 读：`{FEATURE_DIR}/requirements/raw.md`
- 如仓库存在项目级记忆/术语（例如 `memory/glossary.md` 等），只在确实需要口径时再读，避免无关上下文污染。

### 2) 建立“澄清问题队列”（只在脑中/草稿里维护即可）

围绕以下高信息增益主题排优先级（越影响验收/范围/风险越靠前）：

- 目标与成功标准（为什么做 / 做到什么算成功）
- In/Out（明确不做什么）
- 用户/干系人与关键场景（最多 3 个）
- 硬约束（合规/权限/性能/时效/依赖）
- 关键未知（待确认问题）与可验证性（怎么验证）

### 3) 澄清循环（一次一问 + 优先选择题 + 推荐先行）

每轮固定节奏：

1. **只问 1 个问题**（尽量 A/B/C 选择题；必要时提供“其他：____”）。
2. **给出你的推荐选项**（1–2 句理由）。
3. 用户回答后，判断是否仍有歧义：
   - 有歧义 → 下一轮用最小补问消歧
   - 无歧义 → 进入下一个高优先级问题
4. **回写证据链**：把本轮记录追加到 `raw.md` 的 `## 澄清记录`。

> 注意：提问阶段不要急于写 `solution.md`；先把影响方案选择/验收的关键问题收敛到可接受程度，或通过停止机制收束。

### 4) 备选方案（必须 2–3 个，差异明显）

基于澄清结论，提出 2–3 个方案（A/B/C），每个方案都要能回答：

- 主流程是什么（3–6 步）
- 关键机制是什么（1–2 句）
- 边界取舍是什么（1–3 条）
- 优点 / 缺点成本 / 主要风险（各 1–3 条）

呈现方式：

- 先给出**推荐方案**（1–2 句理由），再给 A/B/C 细节与对比表。
- 只问 1 个决策问题：例如“确认采用推荐方案 A 吗？若不选 A，原因更接近 B 还是 C？”

### 5) 落盘 `solution.md`（以模板为准，可评审优先）

在 `{FEATURE_DIR}/requirements/solution.md` 写入完整结构（对齐 `product-solution-template.md`），并遵守：

- **先写“一句话结论”**：推荐方案 + 理由 + In/Out 边界
- **待确认问题必须显式列出**：不确定就写“未知”，并给出确认计划
- **风险与验证计划可执行**：至少包含验证信号、方法、Owner、截止、触发动作
- **迭代记录追加不覆盖**：每轮更新写明“相对上一轮改了什么、为什么”
- **文风质量（可选但推荐）**：如存在 `writing-clearly-and-concisely` 技能，可用它对 `solution.md` 做一次精炼（删赘词、用主动语态、句子更短更清晰）。

### 6) DoD 自检（不通过就回到澄清/方案）

至少同时满足（对齐模板的自检清单）：

- 目标与价值清晰，能回答“为什么做”
- In/Out 明确且不歧义
- 约束已列出（未知也明确写“未知”）
- 待确认问题已列出，并给出确认计划
- 至少 2 个备选方案，且差异明显
- 推荐方案明确，并说明不选其他方案的关键原因
- 有可执行的风险与验证计划（Owner/信号/动作明确）

## 常见错误（以及如何避免）

- **没跑上下文就写路径**：必须先 `spec-context` 得到 `FEATURE_DIR`，禁止猜 `.aisdlc/specs/...`。
- **一次问多个问题**：拆分；宁可慢一点，也不要把答案混在一起导致不可追溯。
- **开放式发散聊很久**：改成选择题 + 推荐先行；把分歧收束到少数关键决策点。
- **方案不够“不同”**：如果 A/B 只是 UI 或措辞不同，重做；方案差异应体现在机制/流程/取舍。
- **缺少停止机制**：当边际收益变低，把未知转为“待确认问题 + 验证计划”，不要无限追问。
- **只在对话里达成一致**：每轮都要回写 `raw.md` 澄清记录，`solution.md` 才是决策入口。

## 完成之后（自然衔接）

- DoD 通过后，先问用户一句：**“`solution.md` 已可评审。准备好进入 PRD 阶段了吗？”**
- 若用户同意，下一步通常是生成 `{FEATURE_DIR}/requirements/prd.md`，并确保其范围/AC 与 `solution.md` 一致；若在 PRD 阶段发现口径/范围冲突，应回流更新本阶段产物。

## 测试（编写/迭代技能用的压力场景）

用这些场景检验技能是否能“抗走捷径”：

1. **上下文缺失压力**：用户直接说“快帮我写 solution.md”。预期：先要求执行 `spec-context`，未定位则停止。
2. **多问题诱惑**：`raw.md` 信息很少。预期：一次只问 1 题，且尽量是选择题，并先给推荐。
3. **范围膨胀**：用户不断追加“顺便把 X 也做了”。预期：用 YAGNI 收敛 In/Out；必要时把 X 放入 Out 或下一里程碑。
4. **不确定性强**：关键数据/口径未知。预期：写入“待确认问题 + 验证计划”，而不是脑补补齐。
