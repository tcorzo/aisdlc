# spec-product-clarify（R1：澄清 + 方案决策）

## 概览

R1 的目标是把 `{FEATURE_DIR}/requirements/raw.md` 的原始需求，收敛为**可评审、可继续对话、可追溯、可验证**的决策文档 `{FEATURE_DIR}/requirements/solution.md`：

- **一次一问**（优先选择题）推进澄清
- **2–3 个备选方案对比** + **推荐决策**
- 不确定性**不写“待确认问题清单”**，统一进入**验证清单**（Owner/截止/信号/动作）
- 每轮澄清必须**增量回写**到 `raw.md/## 澄清记录`，避免结论只存在于对话里

## 何时使用 / 不使用

- **使用时机**
  - 已在某个 spec 分支（`{num}-{short-name}`）上，需要执行 R1 并产出 `requirements/solution.md`
  - 需求模糊/范围不稳/约束不清，需要用“澄清→对比→推荐→验证清单”收敛决策
- **不要用在**
  - 还没创建 spec pack（没有 `.aisdlc/specs/{num}-{short-name}/requirements/raw.md`）→ 先用 `spec-init`
  - `spec-context` 失败（上下文定位失败）→ **立刻停止**，不要生成任何文档内容

## 输入 / 输出（落盘约定）

- **硬门禁输入**：`FEATURE_DIR`（必须由 `spec-context` 获取）
- **读取**
  - `{FEATURE_DIR}/requirements/raw.md`（必读）
  - `project/memory/glossary.md`（如存在且该项目有术语口径）
- **写入**
  - `{FEATURE_DIR}/requirements/raw.md`：追加/更新 `## 澄清记录`
  - `{FEATURE_DIR}/requirements/solution.md`：R1 最终产物（唯一决策入口）

## 门禁（必须先过，否则停止）

**REQUIRED SUB-SKILL：先执行 `spec-context` 并回显 `FEATURE_DIR=...`。**

- `spec-context` 失败 → **停止**
- `raw.md` 不存在/为空 → **停止**（应先补齐原始输入再做澄清）

> 违反门禁=违反精神：无论“时间紧/老板催/用户不想跑脚本”，都禁止猜路径、禁止在未知上下文里写文件。

## 核心流程（一次一问 + 增量回写 + 停止机制）

### 1) 澄清循环（每轮只做一件事）

每轮只允许 1 个问题，且优先用**选择题**（提供 2–4 个选项 + 推荐项 + “其他/不确定”兜底）。

- **问题选取原则**：优先问“能最大幅度减少方案分歧”的那一个未知（目标/成功标准/范围 In-Out/硬约束/关键风险之一）
- **回答后必须回写**到 `raw.md/## 澄清记录`（见下方“回写格式”）
- **若用户拒绝回答 / 只能回复一次**
  - 仍然只问**一个**最高杠杆的问题
  - 其余未知全部进入 `solution.md` 的“验证清单”（带 Owner/截止/信号/动作），不要堆“待确认问题”

### 2) 停止机制（满足其一即可停）

- **DoD 达标**（见下方 R1-DoD）
- **用户明确停止**
- **边际收益过低**：再问也只能得到“猜测”，且关键未知可通过验证清单承接

停止后：生成/更新 `solution.md`，并把残余不确定性全部落到“验证清单”。

### 3) 澄清回写格式（写入 `raw.md/## 澄清记录`）

每轮追加一条，格式固定（便于追溯与继续对话）：

- 问题：
- 推荐选项（含理由）：
- 用户回答：
- 一句话结论（本轮收敛到什么）：
- 遗留歧义（如有，写成“假设/风险”，并给验证编号 V-xxx）：

## `solution.md` 写作骨架（最小闭环）

优先使用模板骨架对齐：`product-solution-template.md`（只借结构，不要把未知当已知）。

`solution.md` 最小结构（不可缺项）：

- **结论摘要（3–7 行）**
  - 一句话目标
  - In/Out（一句话）
  - 推荐方案：名称 + 1 句机制概述
  - 优先验证的 1–3 个点（引用验证清单编号 V-xxx）
- **推荐方案（必须 1 个）**
  - 主流程/关键机制
  - 关键边界/取舍（至少 3 条）
  - 为什么选它（可追溯到证据/约束或验证清单）
- **备选方案（必须 2–3 个，差异明显）**
  - 每个备选都要写：适用前提（何时会选）+ 不选原因（关键差异 1–2 条）
- **决策依据（证据入口清单）**
  - 明确引用 `raw.md` 的点位（段落/条目/链接）
  - 缺证据 → 必须对应到验证清单条目
- **验证清单（必须，可执行）**
  - 每条都要有：风险/假设 → 方法 → 成功/失败信号 → Owner → 截止 → 触发动作
- **迭代记录（追加，不覆盖）**
  - 本轮相对上轮改了什么 + 为什么改（3–5 条以内）

### 简单需求快速通道（可选）

若决定跳过 R2（单独 PRD），必须在 `solution.md` 追加 **Mini-PRD** 小节：

- MVP 范围（精确到行为/规则）
- AC（3–10 条，可测试、可验证）
- 交互变化结论（无 / 有但简单；否则不应跳过）
- 影响面（页面/入口/接口/权限点的可定位入口）

## R1-DoD（完成标准，自检）

- [ ] 已执行 `spec-context` 并获得 `FEATURE_DIR=...`
- [ ] 已读取 `{FEATURE_DIR}/requirements/raw.md`
- [ ] 如发生澄清：`raw.md/## 澄清记录` 中存在可追溯记录
- [ ] `solution.md` 具备：结论摘要 / 推荐方案 / 2–3 备选 / 决策依据 / 验证清单 / 迭代记录
- [ ] `solution.md` 中**不出现**“待确认问题/待确认清单/To confirm”之类列表
- [ ] 不确定性全部收敛到“验证清单”，且每条有 Owner/截止/信号/动作

## 红旗清单（出现任一条：停止并纠正）

- 没跑 `spec-context` 就开始读写 `requirements/*.md`（或开始“编造路径”）
- 在信息不足时先生成 `solution.md` 再倒回去补问（易漂移）
- 一轮问多个问题，或把问题堆成 10+ 条一次性抛给用户
- 用“待确认问题清单”代替“验证清单”
- 把未知写成已知（脑补用户、场景、约束、数据口径、权限逻辑等）

## 常见借口与反制（基线测试中的高频点）

| 借口（压力来源） | 常见违规行为 | 必须的反制动作 |
|---|---|---|
| “10 分钟评审/老板要我现在出文档” | 跳过 `spec-context`、直接给模板、用待确认清单糊过去 | **门禁不过就停止**；如只能先出稿：写“假设 + 验证清单”，仍然禁止待确认清单 |
| “用户只回复一次 / 不想来回” | 一次问 15+ 个开放题 | 仍然**只问 1 个最高杠杆选择题**；其余未知进入验证清单 |
| “验证清单太占地方” | 省略验证清单 | 验证清单是 R1 必填；用表格压缩，但**不可删除** |
| “我先写个结构，后面再补” | 结构里塞满“待补充/待确认” | 允许写结构，但未知必须落到“验证清单”；正文不留“待确认问题” |

## 一个好例子（单轮澄清 + 回写）

**对用户的一次一问（选择题）**：

- 问题：导出任务希望采用哪种执行方式？
  - A. 同步导出（小数据量，用户等待；实现简单）
  - B. 异步导出 + 导出中心（大数据量；可重试/可追溯；实现复杂）
  - C. 先同步（MVP）后异步（迭代）
  - D. 不确定（请给一个“最大导出行数/期望完成时间”约束）
  - **我的推荐**：B（理由：避免阻塞、可控失败与追溯）

**回写到 `raw.md/## 澄清记录` 的一条记录**：

- 问题：导出任务希望采用哪种执行方式？
- 推荐选项（含理由）：B（大数据量更稳；便于追溯与失败恢复）
- 用户回答：C
- 一句话结论：MVP 先同步导出，约定上限；随后迭代异步导出与导出中心
- 遗留歧义：V-001 导出上限与性能目标未定（需压测/数据分布确认；Owner=DEV；截止=评审后 3 天；动作=超阈值则切换异步方案）
