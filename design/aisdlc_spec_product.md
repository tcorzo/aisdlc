## 产品设计流程

### 1. 背景与目标（对齐 `design/aisdlc.md`）

本文档定义 **Spec 级**需求分析流程：在遵循“双层 SSOT + Spec as Code + 渐进式披露”的前提下，用 **可拆分、可独立优化的 AI 对话模块**，把模糊需求推进到以下产物：

- **PRD**：可评审、可验收、可拆解、可追溯
- **原型**：低/中保真均可；在 `prototype.md` 中落盘任务流、页面结构、节点编号与 AC 映射，可进入 UI 设计与研发

> 说明：需求分析阶段的主落盘目录为 `requirements/`（PRD/用例/验收口径等）。交互与原型也可放在 `design/`，取决于团队分工（建议见 2.1）。

---

### 2. 产物落盘与“渐进式披露”的读取顺序

#### 2.1 推荐落盘结构（需求级 Spec Pack）

对齐 `design/aisdlc.md` 中的需求级目录（示例）：

```text
.aisdlc/specs/<DEMAND-ID>/
  index.md
  requirements/
    raw.md                    # 原始输入（工单/访谈/截图/会议纪要摘要）
    solution.md               # 最终方案（澄清结论 + 备选方案 + 推荐方案 + 风险/验证 + 对照证据（可选）+ 迭代记录）
    prd.md                    # PRD（可评审版本）
    prototype.md              # 原型说明（交互内容：任务流/页面结构/节点/AC 映射 + 页面清单、线框、交互细节、状态/校验）
  design/
    architecture.md           # 如涉及架构/契约/ADR，再下沉到 design/
```


#### 2.2 Agent 读取顺序（渐进式披露）

- **必读（项目级）**：先读项目级 `memory/`（业务、技术、结构、术语）与相关 contracts/adr 索引，避免把业务口径、约束、边界写错。
- **按需（需求级）**：仅在明确处理某个 `<DEMAND-ID>` 时读取该需求的 `requirements/*`，并尽量只读最小必要内容。
- **回写（入库）**：每个模块独立产出一个文件（或一个章节），保持可替换与可优化。

#### 2.3 上下文自动识别机制

对齐 `design/aisdlc_spec_init.md` 的“上下文自动识别机制”：所有 spec 级 Skill 执行前先按标准流程获取当前 spec 的上下文信息。

**要做什么**：
- 调用脚本函数 `Get-SpecContext`（位于 `.aisdlc-cli/scirpts/spec-common.ps1`）
- 该函数会：
  - 获取当前 Git 分支名称（`git branch --show-current`）
  - 基于分支名称 `{num}-{short-name}` 构建 Spec 目录路径 `.aisdlc/specs/{num}-{short-name}/`
  - 验证分支格式和目录结构

**输出什么**：
- **REPO_ROOT**：Git 仓库根目录路径
- **CURRENT_BRANCH**：当前分支名称（格式：`{num}-{short-name}`）
- **FEATURE_DIR**：当前需求的 Spec Pack 根目录路径（`.aisdlc/specs/{CURRENT_BRANCH}/`）
- **SPEC_NUMBER**：分支编号部分（从分支名称提取）
- **SHORT_NAME**：分支短名称部分（从分支名称提取）

所有模块（R1-R3）的输入文件路径基于 `FEATURE_DIR` 自动构建，例如：
- `requirements/raw.md` → `{FEATURE_DIR}/requirements/raw.md`
- `requirements/solution.md` → `{FEATURE_DIR}/requirements/solution.md`
- `requirements/prd.md` → `{FEATURE_DIR}/requirements/prd.md`
- `requirements/prototype.md` → `{FEATURE_DIR}/requirements/prototype.md`

### 3. 分布式流程总览（模块化流水线）

目标：每一步都能单独执行、单独评审、单独替换；同时步骤之间通过明确的输入/输出对齐，避免重复劳动与漂移。

#### 3.1 模块清单

| 模块 ID | 模块名称 | 主要目标 | 关键输出 |
|---|---|---|---|
| R0 | Spec 初始化（创建工作分支与 Spec Pack） | 建立“需求上下文”（分支+目录）并落盘原始需求，为后续模块提供稳定的输入路径与自动识别能力 | Spec 分支 `{num}-{short-name}` + `.aisdlc/specs/{num}-{short-name}/` + `requirements/raw.md` |
| R1 | Brainstorming：想法 → 最终方案 | 用“一次一问 + 备选方案对比 + 渐进式验证”的对话，把模糊需求稳定落盘为可评审方案 | `requirements/solution.md` |
| R2 | 基于方案生成 PRD | 把决策转为可交付规格（含 AC） | `requirements/prd.md` |
| R3 | 原型生成（交互 + 线框） | 把 PRD 的场景转为任务流/页面结构/节点编号/AC 映射，并输出可评审的**线框原型**与状态说明 | `requirements/prototype.md` |

#### 3.2 Skill 工作流（每个节点一个 Skill，执行后给出下一步）

本流程的**操作粒度**是“一个节点 = 一个 Skill = 一个落盘产物”。为了让不熟悉流程的同学也能快速上手，建议把 Skill 当作“路标”使用：**每执行完一个 Skill，立即根据本节的引导执行下一个 Skill**。

**Skill 与模块对照**（Skill 定义文档是实现约束 SSOT；主文档是端到端流程 SSOT）：

| 模块 | 使用的 Skill（Skill 定义文件） | 产物（落盘） | 下一步（建议） |
|---|---|---|---|
| R0 | `spec-init`（`.cursor/skills/spec-init/SKILL.md`） | `requirements/raw.md` | → R1：`spec-product-clarify` |
| R1 | `spec-product-clarify`（`.cursor/skills/spec-product-clarify/SKILL.md`） | `requirements/solution.md` | → R2：`spec-product-prd` |
| R2 | `spec-product-prd`（`.cursor/skills/spec-product-prd/SKILL.md`） | `requirements/prd.md` | → R3：`spec-product-prototype` |
| R3 | `spec-product-prototype`（`.cursor/skills/spec-product-prototype/SKILL.md`） | `requirements/prototype.md` | → 原型评审/最小可用性验证 → 回流更新 R1/R2/R3 |

> 说明：Skill 的规范定义以 `.cursor/skills/` 下对应 `SKILL.md` 为准；本文档负责端到端总览与落盘约定。

#### 3.3 最短路径（建议先跑通）

`spec-init` → `spec-product-clarify` → `spec-product-prd` → `spec-product-prototype`

### 4. 模块 R0：Spec 初始化（创建工作分支与 Spec Pack）

#### 4.1 目标

把“原始需求输入”快速转为**可追溯、可自动识别上下文**的 Spec 工作空间：通过创建 `{num}-{short-name}` 分支与标准化 Spec Pack 目录，为后续 R1-R3 提供稳定的文件路径与一致的执行上下文。

#### 4.2 输入与触发

- **输入**：原始需求文本或原始需求文件路径（优先文件路径，以降低中文参数传递的编码风险）。
- **触发时机**：当还没有对应的 spec 分支/目录，或需要开启一个全新的需求 Spec Pack 时执行。

#### 4.3 输出（概要）

- **分支**：创建并切换到 `{num}-{short-name}`（编号自增，命名规范统一）。
- **目录**：创建 `.aisdlc/specs/{num}-{short-name}/` 及标准子目录（`requirements/`、`design/`、`implementation/`、`verification/`、`release/`）。
- **原始需求落盘**：写入 `requirements/raw.md`（作为后续所有分析与设计的“证据入口”）。

#### 4.4 与“上下文自动识别”的关系

R0 的核心价值之一是把 **Git 分支**作为需求级上下文标识符：后续模块在执行前通过 `Get-SpecContext` 获取 `FEATURE_DIR`，从而无需额外参数即可定位当前需求的 Spec Pack（详见 `design/aisdlc_spec_init.md`）。

#### 4.5 关键约束（概要）

- **非破坏性**：不覆盖已存在的分支或目录（除非明确确认）。
- **一致性**：分支名与 Spec 目录名一一对应，保证自动识别可靠。
- **编码统一**：中文内容相关文件以 UTF-8 with BOM 写入，减少跨环境乱码风险。

#### 4.6 执行 Skill（R0）

- **Skill**：`spec-init`（见 `.cursor/skills/spec-init/SKILL.md`）
- **用户输入**：建议优先传入“原始需求文件路径”（降低中文参数传递的编码风险）；也支持直接粘贴文本
- **完成后检查**：
  - 当前分支已切换为 `{num}-{short-name}`
  - `.aisdlc/specs/{num}-{short-name}/requirements/raw.md` 已生成（UTF-8 with BOM）

#### 4.7 下一步（R0 → R1）

执行 R0 后，立即进入 R1 做 Brainstorming（澄清 + 方案）：

- **下一步**：`spec-product-clarify`（生成 `requirements/solution.md`）

### 5. 模块 R1：Brainstorming（澄清 + 方案）

#### 5.1 目标

把输入的原始需求加工为可执行的澄清结论与可评审方案：**明确要做什么、为什么做、对谁有价值、边界是什么，以及推荐怎么做**。

#### 5.2 输入

**上下文自动获取**：
- 通过上下文自动识别机制获取 `FEATURE_DIR`
- 自动定位 `{FEATURE_DIR}/requirements/raw.md`（尽量保留原始措辞、证据链接/截图）

**项目级资源**：
- 项目级 `memory/glossary.md`（术语与口径，如有）

#### 5.3 输出（落盘到 `requirements/solution.md`）
`solution.md` 是本流程的**单一决策入口**：把“澄清结论 + 备选方案 + 推荐方案 + 风险/验证”写入一份可评审文档里，避免在多个文件之间来回对齐。

核心内容（按需扩展）：

- **需求澄清**：
  - **设计目的**：为什么要做？核心价值是什么？
  - **目标用户/干系人**：谁遇到问题？谁使用？谁决策/验收？（简单需求可统一为用户）
  - **关键场景**：用“情景五要素”描述（要素罗列/情景标题/情景描述/痛点与收益点/功能启发）
  - **目标与非目标**（In/Out）：这次做什么？明确不做什么？
  - **约束**：合规/权限/性能/时效等硬约束（如有）
  - **待确认问题**：影响方案选择或验收的关键未知（必须显式列出）
- **备选方案（必须 2–3 个）**：每个方案写清主流程/关键机制/边界取舍，并给出优缺点
- **推荐方案（必须 1 个）**：给出推荐理由（1–2 句）与不选其他方案的关键原因
- **风险与验证**：关键风险/假设 + 可执行的验证计划（Owner/信号/动作）
- **对照证据（可选）**：如需行业对照或机制启发，可补充竞品/替代方案的证据入口与摘要（链接/截图/记录点位）
- **迭代记录**：多轮追加，不覆盖历史（本轮相对上轮改了什么）

> **扩展场景**：对于复杂需求，可在“关键场景”中补充“情景描述”（故事化）、“痛点与收益点”、“功能启发”等细节；对于简单需求，保持上述核心项即可。

#### 5.4 R1 对话流程（澄清 → 方案 → 设计呈现）

R1 是需求分析中的关键步骤，常见问题是：信息不足、结论只存在于对话、追问无止境、方案与验收脱节。本流程采用“**一次一问 + 备选方案对比 + 渐进式验证 + 增量回写 raw.md**”，在保证推进效率的同时确保可追溯。

**理解想法（澄清）**：

- 先查看当前项目状态（相关 `memory/`、契约/ADR 索引、最近提交）与 `{FEATURE_DIR}/requirements/raw.md`
- 逐个提问以完善想法（**每条消息只问一个问题**；需要深入就拆成多个问题）
- 在可能的情况下优先使用选择题；开放式也可
- 聚焦理解：目的、约束、成功标准（以及影响验收/范围/风险的关键未知）

**探索方案（对比 + 推荐）**：

- 提出 **2–3 种**不同方案及利弊（差异必须明显）
- 以对话方式呈现选项，并给出你的**推荐方案**与理由
- 先给出推荐选项并解释原因（让用户更快做决策/纠偏）

**呈现设计（渐进式验证）**：

- 当你认为已理解要构建的内容后，开始呈现设计
- 拆分为 **200–300 字**的段落；**每段之后询问是否正确**，确认后再进入下一段
- 覆盖（按需取舍）：架构、组件、数据流、错误处理、测试
- 若出现不清晰之处，随时回到澄清（继续“一次一问”）

**落盘与可追溯（建议做法）**：

1. **生成候选澄清问题队列**：结合 `raw.md` 与模板要求（`.aisdlc-cli/templates/spec/product-solution-template.md`）生成“高信息增益”问题，并按影响验收/合规/范围/成本排序；队列可随回答动态增删。
2. **顺序提问循环（一次一个问题）**：
   - 每次只问 1 个问题（优先选择题）
   - 在显著位置给出你的“推荐选项”并说明 1～2 句理由
   - 用户回答后判断是否已澄清；如仍有歧义，用最小补问消歧或调整队列
   - 具备停止机制：DoD 达标 / 用户停止 / 边际收益过低 / 风险可被“待确认问题 + 验证计划”承接
3. **每次接受答案后的集成（增量更新）**：把“问题 + 推荐选项 + 用户回答 + 一句话结论（+遗留歧义，如有）”追加到 `raw.md` 的“澄清记录”，形成证据链。
4. **重新读取更新后的 raw.md**：生成 `requirements/solution.md`（含备选方案、推荐方案、风险与验证、迭代记录）。
5. **验证（最后一遍执行）**：做记录完整性、一致性、模板可填充性与 R1-DoD 自检。


#### 5.5 AI 对话提问脚本（核心 3 问）

1. **目标与价值**：请用一句话复述需求目标，并说明为什么要做（用户目标 + 业务目标）。
2. **用户与场景**：谁遇到问题？在什么情况下遇到？期望如何解决？
3. **边界与约束**：这次明确做什么？不做什么？有哪些硬约束（合规/权限/时效等）？

> **按需扩展**：对于复杂需求，可追加干系人优先级、量化证据、验收标准等问题；对于简单需求，上述 3 问足够。

#### 5.6 质量门槛（R1-DoD）

- 目标与价值清晰（能回答“为什么做”）
- 用户与场景可复述且不歧义
- In/Out 明确（避免范围漂移）
- 约束已列出（未知也显式写“未知”）
- 待确认问题已列出（如有）
- 有证据/引用入口（raw/glossary 等，如适用）
- 有 2–3 个备选方案与推荐方案（体现权衡）
- 有可执行的风险与验证计划（Owner/信号/动作明确）

#### 5.7 执行 Skill（R1）

- **Skill**：`spec-product-clarify`（见 `.cursor/skills/spec-product-clarify/SKILL.md`）
- **门禁（必须先过）**：必须先完成上下文识别并读取 `{FEATURE_DIR}/requirements/raw.md`；如信息不足，按 5.4 的“一次一问”节奏澄清到满足 DoD 或触发停止机制，再生成 `solution.md`（不得先生成再补问导致漂移）。
  - 对用户 **一次只问 1 题**
  - 每轮回答后必须把“问题 + 推荐选项 + 用户回答 + 一句话结论 + 遗留歧义（如有）”**增量写回 `raw.md` 的 `## 澄清记录`**
  - 具备停止机制（DoD 达标 / 用户停止 / 边际收益过低；其余关键未知进入“待确认问题 + 验证计划”）
- **完成后检查**：
  - `requirements/solution.md` 已生成且满足 R1-DoD
  - 若发生澄清问答：`requirements/raw.md` 中存在可追溯的“澄清记录”

#### 5.8 下一步（R1 → R2）

完成 R1 后，进入 PRD 转写阶段：

- **下一步**：`spec-product-prd`（生成 `requirements/prd.md`）

如需要行业对照或机制启发，建议在 `solution.md` 的“对照证据”中记录：证据入口、关键机制摘要、适用前提与关键权衡。

### 7. 模块 R2：基于方案生成 PRD

#### 7.1 目标

把决策方案转为可交付规格：范围清晰、验收可执行、可拆解为研发任务，并能与后续用例/测试/发布追溯。

#### 7.2 输入

**上下文自动获取**：
- 通过上下文自动识别机制获取 `FEATURE_DIR`
- 自动定位 `{FEATURE_DIR}/requirements/solution.md`（含澄清结论、推荐方案、风险与验证计划）

**项目级资源**：
- 项目级术语/约束/契约索引（如涉及）

#### 7.3 输出（落盘到 `requirements/prd.md`）

建议最小结构（可直接作为模板）：

- 背景与问题（来自 R1）
- 目标（用户目标/业务目标）与成功指标（可埋点/可观测）
- 范围（In/Out）与里程碑（MVP→迭代）
- 功能列表与优先级：MoSCoW（或 P0/P1/P2），并与里程碑对齐（MVP 至少覆盖 Must/P0）
- 核心场景与用户故事（按场景组织）
- 业务规则与口径（引用 glossary/数据口径）
- 交互要点（只写关键原则与约束，细节下沉 R3 的 `prototype.md`）
- 异常与边界（失败、权限、并发、幂等、降级）
- 验收标准（AC）：逐条可测试、可验证
- 关键干系人验收口径：高优先级干系人的关注点如何被满足/如何验收（避免“做完没人认”）
- 风险与依赖（含缓解）
- 追溯链接：引用 `solution.md`、后续 `prototype.md`

#### 7.4 质量门槛（R2-DoD）

- 每个核心场景都有 AC（可转测试用例）
- In/Out 与 R1 一致，且与最终方案一致
- 风险与依赖有明确 Owner 与缓解动作
- PRD 的范围/里程碑与功能优先级一致：MVP 至少覆盖 Must/P0，并且 Won't/Out 有明确口径
- 至少覆盖 1 个高优先级干系人的验收口径（能写成可执行的检查项或验收清单）

#### 7.5 执行 Skill（R2）

- **Skill**：`spec-product-prd`（见 `.cursor/skills/spec-product-prd/SKILL.md`）

### 9. 模块 R3：基于 PRD 生成原型（交互 + 线框）

#### 9.1 目标

把 PRD 的核心场景/规则/AC 加工为可评审的原型资产（交互 + 线框），即使不使用 Figma 也能让研发/测试准确理解任务流、页面结构与状态。

#### 9.2 输入

**上下文自动获取**：
- 通过上下文自动识别机制获取 `FEATURE_DIR`
- 自动定位 `{FEATURE_DIR}/requirements/prd.md`（核心场景、规则、AC）

#### 9.3 输出（落盘到 `requirements/prototype.md`）

`prototype.md` 作为“交互内容 + 原型”一体的落盘文件，推荐结构如下（可被后续工具再加工为可点击原型）：

- **交互内容**：
  - 场景清单（与 PRD 对齐）
  - 每个场景的情景摘要（来自 R1 的 `solution.md`）：情景标题/情景描述/痛点与收益点
  - 端到端用户任务流（Mermaid 流程图优先）
  - 页面/弹窗/组件清单（信息架构：从入口到完成）
  - 关键交互细节：空/加载/错/无权限/降级、校验规则与提示文案、风险提示与二次确认
  - 交互策略（绑定痛点/收益点）：关键节点说明对应的设计策略
  - **AC → 交互节点映射表**（能回答“在哪验证哪条 AC”）

- 页面编号与清单（P-001…）
- 每个页面：
  - 页面目的与入口
  - 布局分区（Header/Body/Footer/侧栏等）
  - 信息与控件（字段、按钮、默认值、禁用条件）
  - 状态机（正常/加载/空/错误/权限不足）
  - 跳转与交互（点击/提交/返回/取消/二次确认）
  - 埋点建议（与成功指标对齐，若需要）
- 关键页面线框（可用 ASCII/表格描述，或链接到外部原型文件）
- 最小可用性验证（建议写成固定章节，便于回流）：
  - 测试目标：验证哪些关键假设/痛点是否被解决
  - 测试对象：目标用户（优先）或内部代理（必要时），以及覆盖到的关键干系人视角
  - 任务脚本：按“情景标题→任务目标→成功标准→观察点/记录项”列出
  - 记录方式：问题清单、严重度、复现步骤、建议修复方向
  - 结论与回流：需要回流更新哪些文件（R1/R2/R3），以及为什么

> 若团队使用 Figma：`prototype.md` 仍建议保留页面清单、状态说明、关键规则，并在顶部附上 Figma 链接作为入口，避免原型与规则漂移。

#### 9.4 质量门槛（R3-DoD）

- 交互内容与 PRD 的场景/规则/AC 一一对应
- 任务流、节点清单与页面清单一致（可追溯、可定位）
- 每个页面至少包含：入口、主要控件、状态、跳转
- 与 PRD 的 AC 可追溯（能指出“哪个页面/哪个状态支持哪条 AC”）
- 至少包含一份“最小可用性验证”章节（含任务脚本与回流指引），确保原型验证能形成闭环而非一次性产物

#### 9.5 执行 Skill（R3）

- **Skill**：`spec-product-prototype`（见 `.cursor/skills/spec-product-prototype/SKILL.md`）
- **门禁（必须先过）**：必须存在 `requirements/prd.md`；缺失不得生成 `prototype.md`

#### 9.6 下一步（R3 → 评审验证 → 回流闭环）

R3 结束不是终点，建议立刻做一次最小评审闭环：

- **下一步（评审/验证）**：基于 `prototype.md` 组织原型评审与最小可用性验证
- **发现问题如何回流**（按影响优先）：
  - 口径/边界/场景/方案变化 → 回流 R1（`solution.md`）
  - AC 不可验收/范围冲突 → 回流 R2（`prd.md`）
  - 线框页面结构/内容缺失 → 回流 R3（`prototype.md`）

### 10. 端到端“最小可用”的执行方式（建议）

为了让流程可分布、可迭代，建议采用下面节奏：

- **先做 R1**：产出 `solution.md` 即可评审（不等竞品/方案）
- **R1 内完成方案**：在 `solution.md` 中快速产出 2–3 个机制差异明显的备选方案，并完成推荐决策与验证计划
- **R2 产 PRD**：先产 MVP 版本 PRD（可评审），再迭代补齐细节
- **R3**：围绕核心场景先出关键链路交互与线框原型；有重大问题则回流更新 R1/R2/R3，再继续扩展边缘场景
